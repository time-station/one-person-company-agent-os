# Contributing  
  
欢迎来到这个项目。  
  
这个仓库不仅欢迎代码贡献，也欢迎：  
  
- 方法框架改进  
- 协议优化  
- 模板补充  
- use case 样板  
- 失败经验复盘  
- 社区结构建议  
- 工具原型与实现想法  
- 文档优化与表达修订  
  
我们希望把这里建设成一个围绕**一人制公司与 Agent 智能能力结合**的开源创新共同体。  
  
---  
  
## 我们欢迎什么样的贡献  
  
我们尤其欢迎这些类型的贡献：  
  
### 1. 结构性贡献  
能够帮助项目更清楚、更稳、更可运行的贡献。  
  
例如：  
- 新的协议建议  
- 工作流修订  
- 主经营智能体边界细化  
- 七核角色职责优化  
  
### 2. 实践性贡献  
来自真实使用和真实场景的反馈。  
  
例如：  
- 你用模板跑过一轮任务  
- 你在研究/内容/项目推进中发现了真实问题  
- 你有明确的高耗低值环节观察  
  
### 3. 样板型贡献  
能帮助后来者快速理解“这套系统怎么用”的真实样板。  
  
例如：  
- 内容工作室变体  
- 研究工作室变体  
- 一人咨询/服务流程样板  
- 小团队协同样板  
  
### 4. 表达与文档贡献  
能让项目更容易进入、理解和传播的贡献。  
  
例如：  
- README 改进  
- INDEX 改进  
- 社区导读  
- 说明文档修订  
  
---  
  
## 我们建议的参与方式  
  
如果你是第一次参与，建议按这个顺序：  
  
1. 先阅读 `README.md`  
2. 再看 `INDEX.md`  
3. 再看 `manifesto/open-source-community-charter-v1.md`  
4. 选择一个最接近你的 use case 或模板开始实践  
5. 把你的问题、建议或经验带回 Discussions  
  
---  
  
## 提 issue 前，先想清楚这几件事  
  
在提 issue 或建议前，尽量先回答：  
  
- 你讨论的到底是什么问题  
- 这个问题为什么重要  
- 它和现有结构有什么关系  
- 你希望修正的是哪一个文件、模板、协议或 use case  
- 它属于概念问题、流程问题、样板问题，还是实现问题  
  
这样会让讨论效率高很多。  
  
---  
  
## Pull Request 建议  
  
如果你提交 PR，建议尽量做到：  
  
- 改动目标清楚  
- 不混入无关修改  
- 文件命名保持统一  
- 结构保持克制  
- 尽量解释你为什么这么改  
- 如果涉及主线定义，说明它与现有骨架的关系  
  
---  
  
## 命名与格式建议  
  
当前仓库尽量遵循这些命名规则：  
  
- 文件使用英文小写  
- 单词之间用连字符 `-`  
- 使用版本尾缀 `-v1`  
- 使用 `.md` 文件格式  
  
例如：  
- `chief-operating-agent-definition-v1.md`  
- `task-intake-template-v1.md`  
  
---  
  
## 我们更看重什么  
  
我们更看重：  
  
- 判断  
- 结构  
- 边界  
- 可运行性  
- 真实实践  
- 复盘能力  
- 长期资产沉淀  
  
我们不太鼓励：  
  
- 空泛热闹  
- 无边界大词堆砌  
- 与主线弱相关的热点搬运  
- 为了复杂而复杂  
  
---  
  
## 一句话  
  
**欢迎贡献，但更欢迎能让这套系统更清楚、更真实、更可运行的贡献。**
