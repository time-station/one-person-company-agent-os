# One-Person Project Operations v1

## 一人项目推进系统样板

这是一个最贴近现实创业与建设过程的样板。

如果说内容工作室更适合承接表达，  
研究工作室更适合承接判断，  
那么项目推进系统，就是最适合承接：

- 目标推进
- 阶段拆解
- 节点管理
- 协作调度
- 执行留痕
- 风险上浮
- 复盘沉淀

也就是说，它非常适合把：

**主理人  
→ 主经营智能体  
→ 七核岗位系统**

真正带入持续推进的现实任务中。

---

## 一、这个样板在解决什么问题

很多一人主理人推进项目时，最常见的问题不是没有想法，  
而是：

- 事情太多，不知道先推哪个
- 项目推进全靠临场记忆
- 阶段没有拆清，导致总在卡住
- 任务之间没有交接逻辑
- 做了很多，但没有形成系统感
- 每次推进都像重新开始

所以，一人项目推进系统的目标不是“做更多项目”，  
而是：

**让项目推进开始具备最小经营秩序。**

---

## 二、样板的基本结构

一人项目推进系统的最小结构是：

**主理人  
→ 主经营智能体  
→ 七核岗位系统  
→ 项目推进闭环**

其中：

### 主理人负责
- 决定项目目标
- 决定优先级
- 决定边界
- 对关键节点拍板
- 对项目最终结果负责

### 主经营智能体负责
- 把项目意图翻译成推进任务
- 拆出阶段与节点
- 调度七核岗位系统
- 管理推进节奏
- 汇总状态
- 暴露风险
- 输出阶段性反馈

### 七核岗位系统负责
- 判断项目现在该不该推进
- 编排阶段与节点
- 做必要研究与补信息
- 整理输出物
- 推动执行
- 审计断链与风险
- 沉淀模板与经验

---

## 三、适合什么样的项目

这个样板最适合：

- 仓库搭建
- 内容产品搭建
- 方法论项目推进
- 研究项目推进
- 小型创业试点
- 产品雏形建设
- 开源项目建设
- 文档系统建设

它特别适合那些：

**任务多、结构多、阶段多、但团队仍然很小的项目。**

---

## 四、最小工作链路

一轮标准项目推进，最小可以这样跑：

### 第一步：项目进入
主理人提出一个明确要推进的项目，例如：

- 建一个开源社区仓库
- 完成第一版方法论骨架
- 推出一个产品试点
- 跑通一个 use case
- 完成一个阶段性发布

项目推进的起点，最好是一个明确对象，  
而不是模糊愿望。

---

### 第二步：判断前置
判断 Agent 先做第一轮判断：

- 这个项目现在该不该启动
- 它是不是当前阶段最高优先级
- 它的主要风险是什么
- 它适合先做最小版本还是直接做完整版本

这一步的价值是：
**避免主理人被热情推动，但没有结构判断。**

---

### 第三步：阶段拆解
编排 Agent 把项目拆成阶段，例如：

- 第一阶段做什么
- 第二阶段做什么
- 哪一部分先立骨架
- 哪一部分先做门面
- 哪一部分以后再补
- 哪些节点必须主理人亲自拍板

没有这一步，项目就很容易陷入：
- 做了很多
- 但没有阶段感
- 也没有推进感

---

### 第四步：研究与信息补齐
研究 Agent 在必要时补齐：

- 相关资料
- 参考案例
- 类似项目
- 可借鉴结构
- 风险点
- 外部生态判断

但项目推进中的研究必须克制。  
研究要服务推进，而不能反客为主。

---

### 第五步：输出物组织
表达 Agent 把阶段成果整理成清楚的输出物，例如：

- README
- 目录结构
- 文档
- 方案稿
- 阶段总结
- 发布文案
- 结构图
- 说明文档

这一层很重要，
因为项目推进不是靠脑中感觉完成，
而是靠一份份输出物被做出来。

---

### 第六步：执行推进
执行 Agent 负责：

- 真正建文件
- 推动节点完成
- 完成上传、发布、更新
- 跟踪当前阶段完成情况
- 推动下一步动作发生

执行层的价值不是“忙”，  
而是：

**让阶段目标真正落地。**

---

### 第七步：监督审计
监督审计 Agent 检查：

- 有没有任务断链
- 有没有关键节点遗漏
- 有没有走偏主线
- 有没有返工风险
- 有没有边界失控
- 有没有阶段目标与现实动作脱节

这一步可以有效减少：
- 一人主理人忙乱推进
- 越做越散
- 最后不知道自己到底推进了什么

---

### 第八步：记忆与沉淀
记忆 Agent 负责把项目推进中的高价值内容沉淀下来，例如：

- 目录结构
- 模板
- 协议
- 版本节点
- 阶段总结
- 常见问题
- 失败教训
- 最小可复用工作流

这一步决定项目是不是在“越做越强”。

---

## 五、主经营智能体在项目推进中的作用

主经营智能体在项目推进里尤其像一个真正的经营代理。

它不是自己完成项目，  
而是：

### 1）把项目从愿望变成推进结构
很多主理人心里知道要做什么，  
但不知道该怎么拆、怎么排、怎么跑。

主经营智能体负责把它结构化。

### 2）把推进节奏组织起来
不是“想到哪做哪”，  
而是：
- 哪一步先做
- 哪一步后做
- 哪一步现在不该做
- 哪一步需要主理人回来拍板

### 3）把状态压缩给主理人
主理人不需要盯全部细节，  
而是通过主经营智能体看到：

- 现在到哪了
- 卡点在哪
- 下一步是什么
- 哪些事情该判断了

### 4）让项目形成经营反馈
每推进一段，不只看：
“做完了吗？”

还要看：
- 值不值得继续
- 哪部分最耗损
- 哪部分最有效
- 哪些东西已经值得沉淀成公共资产

---

## 六、这个样板最容易沉淀的资产

一人项目推进系统特别容易沉淀：

### 1）推进结构资产
哪些阶段拆法最合理。

### 2）目录与文档资产
哪些结构适合反复复用。

### 3）工作流资产
什么样的推进链路最顺。

### 4）判断资产
什么项目该先做，什么该延后。

### 5）节奏资产
你最适合的推进节奏是什么。

### 6）复盘资产
哪些卡点经常出现，怎样提前规避。

---

## 七、这个样板为什么重要

因为很多创业方向最后不是死在“没有灵感”，  
而是死在：

- 推不动
- 推不稳
- 推得太乱
- 推完没有留下资产

一人项目推进系统之所以重要，
是因为它可以让一个小主体逐步长出：

- 节奏
- 结构
- 交接
- 反馈
- 沉淀

这其实就是最小经营感的开始。

---

## 八、最小使用方式

如果今天就要推进一个项目，不需要全部系统都做完。

最小可以这样跑：

1. 主理人明确一个项目目标  
2. 用任务审计卡判断是否现在推进  
3. 拆出三阶段结构  
4. 补齐必要研究与参考  
5. 做出当前阶段输出物  
6. 用执行留痕模板记录推进  
7. 用复盘卡回收阶段反馈  
8. 更新下一步任务清单  

只要这 8 步走通，  
这个样板就已经活起来了。

---

## 九、一句话总结

**一人项目推进系统样板，是把项目建设从“靠个人临场推进”升级为“由主理人掌握方向、由主经营智能体组织经营、由七核岗位系统完成闭环”的最小项目经营系统。**
